# 設計判断の記録

このドキュメントでは、RoboTasker の設計上の重要な判断とその理由を記録します。
ポートフォリオ評価の際に「なぜこの設計にしたか」を示す参考資料です。

---

## 1. なぜ Go をバックエンドに選んだか

**判断**: Laravel (PHP) ではなく Go を採用

**理由**:
- ROS2 との WebSocket 通信に goroutine の並行処理が自然にマッチする
- 単一バイナリでデプロイでき、Docker イメージが軽量 (~15MB)
- ロボット制御システムのバックエンドとしてリアルタイム性が求められる場面で Go のパフォーマンスが有利
- 型安全性が高く、API の信頼性を担保しやすい

**トレードオフ**: Laravel のような ORM やスキャフォールディングがないため、リポジトリパターンを手書き

---

## 2. なぜ rosbridge (WebSocket) で接続したか

**判断**: Go から直接 DDS に接続するのではなく、rosbridge 経由の WebSocket 通信を採用

**理由**:
- Go には成熟した ROS2 DDS クライアントライブラリが存在しない
- rosbridge は ROS2 公式でサポートされており、言語非依存
- WebSocket は Go (gorilla/websocket) と Python (websockets) の両方で扱いやすい
- フロントエンドへのリアルタイム配信にも WebSocket を統一できる

**トレードオフ**: rosbridge のオーバーヘッド（シリアライズ+WebSocket）により、レイテンシが DDS 直接通信より大きい

**将来**: rclgo (Go 用 ROS2 バインディング) が安定したら DDS 直接接続に切り替え可能

---

## 3. なぜ LLM にインターフェースパターンを使ったか

**判断**: `LLMService` インターフェースを定義し、MockLLMService と OpenAI 実装を切り替え可能にした

**理由**:
- ポートフォリオ段階では OpenAI API キーなしでもデモが動く必要がある
- テスト時にモック実装で決定的な結果を返せる
- 将来的に Claude、ローカル LLM (Ollama 等) に切り替える際にサービス層の変更だけで対応

**設計パターン**: Strategy パターン + Dependency Injection

---

## 4. なぜタスク状態遷移をログテーブルに記録したか

**判断**: tasks テーブルの status 更新と同時に task_logs テーブルに遷移履歴を記録

**理由**:
- 業務システムでは監査ログ（audit trail）が求められる
- タスクの異常検知やパフォーマンス分析に遷移時刻が必要
- 「バックエンドエンジニアとして業務システム設計ができる」ことを示す

---

## 5. なぜ bridge_node に ID マッピング機能を持たせたか

**判断**: ROS2 側のローカルロボットID (`robot-01`) とバックエンドの UUID をブリッジノードでマッピング

**理由**:
- ROS2 ノード（navigation_controller 等）は軽量であるべきで、DB の UUID を意識させたくない
- 複数ロボット対応時に、1 bridge_node が複数ロボットの ID を管理できる構造
- バックエンド側は UUID で一貫した操作ができる

**トレードオフ**: タスク送信前はマッピングが存在しないため、アイドル時の位置更新が無視される

---

## 6. なぜモノレポ構成にしたか

**判断**: バックエンド、フロントエンド、ROS2 を1リポジトリで管理

**理由**:
- ポートフォリオとして1つの GitHub リポジトリで全体を俯瞰できる
- docker-compose で全体を一発起動できる
- カスタムメッセージ定義の変更が全レイヤーに影響するため、同一リポジトリだと整合性を保ちやすい

**将来**: プロダクション化する場合は、バックエンド・ROS2・フロントエンドを個別リポジトリに分離

---

## 7. なぜ簡易シミュレーターを先に作ったか

**判断**: Gazebo の前に Python ベースの簡易移動シミュレーター (navigation_controller) を実装

**理由**:
- Gazebo のセットアップは環境依存が大きく、初期段階でボトルネックになる
- 簡易シミュレーターならフロー全体（チャット→タスク作成→移動→完了）を数分で動かせる
- navigation_controller の `/navigation/goal` インターフェースは Nav2 と互換なので、差し替えが容易

---

## 8. フロントエンドの技術選定

**判断**: Next.js + TypeScript + Tailwind CSS

**理由**:
- React の実務経験を活かせる
- App Router + Server Components で構造が明確
- Tailwind CSS でデザインシステムなしでも統一感のある UI を素早く作れる
- SSR/SSG は使わず、全ページ `"use client"` で SPA 的に動作させる（リアルタイム更新のため）

---

## 9. エラーハンドリング方針

**判断**: レイヤーごとに責務を分け、過剰なリトライはしない

| レイヤー | 方針 |
|---|---|
| LLM | confidence < 0.5 で確認メッセージを返す。タイムアウトでモックにフォールバック |
| タスク割当 | ロボット不足なら pending に残す（ユーザーには通知） |
| 移動 | 60秒タイムアウトで failed。自動リトライはせず手動再送 |
| WebSocket | 切断時は自動再接続（バックエンド/フロントエンドともに） |

**理由**: ロボットシステムでは「失敗を隠す」のが最も危険。明示的に失敗を報告し、人間の判断を仰ぐ設計。
